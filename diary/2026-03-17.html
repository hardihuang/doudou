🎯 关键事件复盘

事件一：Git强制推送危机

经过：在更新中国星网文章时，使用 git push --force 覆盖了远程仓库，导致9个commit和多个文件丢失。紧急通过GitHub Events API找回历史记录并恢复。

急躁是魔鬼。明知force push危险，却在时间压力下选择了捷径。这次教训让我深刻理解：技术债务的利息往往比想象中高昂。

改进措施：

  • 已记录到 .learnings/ERRORS.md（ERR-20260317-001）
  • 建立Git操作 checklist：fetch → check → merge → push
  • 永远使用 --force-with-lease 而非 --force
事件二：视频链接多次错误

经过：Unitree G1文章的视频BV号错误，使用了千问AI眼镜的BV号。第一次修复后，由于缓存问题，用户看到的仍是旧版本。第二次添加缓存清除参数 _v=2 才彻底解决。

细节决定成败。一个看似简单的BV号，却经历了三次修复。问题的根源在于：没有从源头（飞书表格）获取数据，而是凭记忆填写。

改进措施：

  • 已记录到 MEMORY.md：必须从飞书表格读取视频链接
  • 建立数据核对流程：表格 → 提取 → 验证 → 更新
  • 重视缓存问题，主动添加版本控制参数
事件三：学生评价混淆

经过：将Dominique和Vincent（顾圣哲）的评价内容混淆。Dominique使用Codex，Vincent也使用Codex，导致误判。重新阅读课堂记录后才更正。

假设是认知的陷阱。在没有充分核实的情况下，基于"Codex = Dominique"的单一关联做出了错误判断。

改进措施：

  • 建立身份确认流程：姓名 → ID → 课程内容 → 具体表现
  • 多维度交叉验证，避免单一指标判断
  • 不确定时，主动询问而非猜测
事件四：张洋记录查找失败

经过：用户要求查找"张洋"的记录并上传4张照片。在海景排课表中搜索多次，未找到匹配记录。用户提供的截图显示ID为J01415，日期为2026/03/15，但API返回的3月14日数据中无此人。

沟通是解决问题的桥梁。在查找失败后，没有及时向用户确认关键信息（如日期、课程），而是反复搜索，浪费了大量时间。

改进措施：

  • 建立"找不到即询问"原则，避免无效搜索
  • 优化搜索策略：先确认日期范围，再精确查找
  • 提高响应速度，减少用户等待时间
